\chapter{Theoretical Background}

\section{Basics of Cryptography} 

Cryptography, derived from Greek words meaning "hidden writing", is the study
and practice of mathematical methods for secure communication over insecure
channels. In modern systems, however, it is no longer limited to
confidentiality alone, but also addresses integrity, authentication, and
non-repudiation. More broadly, modern cryptography is concerned not only with
secrecy, but also with detecting modification and verifying
authenticity.\cite{diffie1976newDirections,ibm2023cryptography}

The need for secret communication predates modern computing by thousands of
years. One early example is the use of unusual hieroglyphs carved into the wall
of a tomb from the Old Kingdom of Egypt, often cited as one of the earliest
known uses of cryptography.~\cite{schneider2024ibmhistory}

Before computers, cryptography relied on classical ciphers that operated on
letters rather than bits. One of the most popular examples is the Caesar
cipher, which shifts each letter by a fixed number of positions in the
alphabet. Because the number of possible shifts is small, the cipher is easy to
break by trying all possibilities.~\cite{aumasson2017serious}

A more advanced classical cipher is the Vigen\`ere cipher, which uses a keyword
to vary the shift applied to each letter. This makes it stronger than the
Caesar cipher, but it remains vulnerable to frequency analysis. Classical
ciphers were historically important, yet their security was too weak for modern
communication systems.~\cite{aumasson2017serious}

As the need for secure communication grew, more sophisticated systems were
developed, including mechanical rotor machines such as the Enigma machine.
These devices used changing substitutions, which made them harder to break than
earlier hand ciphers.~\cite{schneider2024ibmhistory}

Their eventual defeat showed that security could not rely only on increasingly
complicated substitutions. From the 1970s onward, cryptography began to move
toward mathematically defined constructions whose security depended on the
difficulty of underlying computational problems. This shift marks the beginning
of modern cryptography and motivates the discussion in the next section.
~\cite{diffie1976newDirections}

\section{P, NP, and Computational Hardness}

In cryptography, the security of many systems depends on problems that are
believed to be computationally hard to solve. To describe meaning to
computational difficulty, problems are organized into complexity classes. The
groups are based on how the running time of an algorithm grows with the size of
the input. One important class is $\mathcal{P}$, which contains decision problems that
can be solved in polynomial time. A decision problem meaning one with a yes or
a no answer is in $\mathcal{P}$ if there exists an algorithm that solves any instance in
time bounded by a polynomial in the input size. Intuitively, problems in $\mathcal{P}$
are considered efficiently solvable.

Another fundamental class is $\mathcal{NP}$, which stands for nondeterministic
polynomial time. A decision problem belongs to $\mathcal{NP}$ if a proposed
solution, often called a certificate, can be verified in polynomial time.
This means that even if we do not know how to find a solution quickly, we can
check whether a given solution is correct in a reasonable time. 

% It is also
% important to note that $\mathcal{NP}$ does not mean “not polynomial”; instead,
% it refers to the way problems are defined in terms of verifiability.

A major open problem in theoretical computer science is whether
$\mathcal{P} = \mathcal{NP}$. Many researchers believe that $\mathcal{P} \neq
\mathcal{NP}$, but this has not been proven. If it turned out that
$\mathcal{P} = \mathcal{NP}$, then every problem whose solution can be verified
quickly could also be solved quickly. This would have enormous consequences:
many problems currently considered difficult would become easy, and most modern
cryptographic systems would no longer be secure. Because no polynomial-time
algorithm is known for many $\mathcal{NP}$ problems, the prevailing assumption
is that $\mathcal{P}$ $\neq$ $\mathcal{NP}$.

Within the class $\mathcal{NP}$  there exists a subset of problems that are, in a
precise sense, the most computationally difficult problems in the class. These
are referred to as $\mathcal{NP}$-complete problems. A decision problem $C$ is said to
be $\mathcal{NP}$-complete if it satisfies two requirements: it belongs to
$\mathcal{NP}$, and every problem in $\mathcal{NP}$ can be transformed into C by a
polynomial-time reduction. If a polynomial-time algorithm were discovered for
any $\mathcal{NP}$-complete problem, then all problems in $\mathcal{NP}$ would become
solvable in polynomial time. Since no such algorithm is known,
$\mathcal{NP}$-complete problems are generally regarded as computationally
unsolvable. 

One well-known $\mathcal{NP}$-complete problem is the subset-sum problem. In the decision
version of this problem, we are given a set of numbers and a target sum, and we
ask whether there exists a subset of the numbers whose sum equals the target.
It is easy to verify whether a chosen subset achieves the target sum, but
finding the correct subset appears to require exponential time in the worst
case. This contrast between easy verification and difficult search is typical
for $\mathcal{NP}$-complete problems and is one of the reasons why the Knapsack problem has
been used in cryptography.

There is also a broader category known as $\mathcal{NP}$-hard problems. Informally,
$\mathcal{NP}$-hard means that a problem is at least as difficult as any problem in
$\mathcal{NP}$. More formally, a problem $H$ is $\mathcal{NP}$-hard if every problem in
$\mathcal{NP}$ can be reduced to $H$ in polynomial time. $\mathcal{NP}$-hard problems do not
need to be decision problems, and they do not need to have efficiently
verifiable solutions. If an $\mathcal{NP}$-hard problem is also in $\mathcal{NP}$, then it
is $\mathcal{NP}$-complete. In other words, $\mathcal{NP}$-complete problems are exactly those that lie
in the intersection $\mathcal{NP} \cap \mathcal{NP}$-hard.

Modern cryptographic algorithms rely on the assumed hardness of such problems.
The security of a cryptosystem is based on the fact that no efficient algorithm
is known for solving its underlying mathematical problem. For instance, the RSA
cryptosystem is based on the difficulty of integer factorization, which is not
known to be $\mathcal{NP}$-complete but is still considered hard. The classical Merkle--Hellman
knapsack cryptosystem, on the other hand, is built directly on the $\mathcal{NP}$-complete
Knapsack problem. Because solving the general knapsack problem appears to
require exponential time, an attacker without the secret key faces an extremely
difficult computational task. As long as no efficient algorithms are found for
these kinds of hard problems, cryptographic schemes based on them remain secure.

\section{Symmetric and Asymmetric Cryptography}

Modern cryptography is divided into two main types. Symmetric and Asymmetric
also known as public-key cryptography. Aim of both is to secure communication,
but they differ in how the keys are used and mathematical concepts that make
them work giving each approach its own strengths and weaknesses.

Symmetric cryptography follows the same basic idea that classical ciphers were
built on. In this approach, all parties involved in transaction use a single
shared secret key for both encryption and decryption. They must exchange this
key in a secure way before communicating, so key distribution becomes an
important problem.

Because of this limitation, symmetric systems are often not practical when two
parties have never communicated before or do not have a secure channel for
exchanging the key. This problem motivated the development of a different
approach, where the communicating parties do not need to share a secret in
advance. This idea led to asymmetric, or public-key cryptography.

Asymmetric cryptography works with a pair of keys: a public key that can be
shared with anyone and a private key that must remain secret. Messages
encrypted with the public key can only be decrypted with the corresponding
private key, which removes the need for a secure key exchange. The security of
these systems depends on mathematical problems that are easy to compute in one
direction but extremely difficult to reverse without the private key.
...
These ideas form the basis of modern public-key cryptography. Many encryption
schemes rely on specific mathematical problems that are assumed to be hard to
solve without the secret key. One such problem is the subset-sum, also known as
the knapsack problem, which plays a central role in the cryptosystem studied in
this thesis. The following sections introduce this problem and explain how it
can be used to construct a public-key encryption scheme.

\section{The Knapsack Problem}

The knapsack problem, also known as the subset--sum problem, is a fundamental
combinatorial decision problem. Given a sequence of positive integers
\[
W = (w_1, w_2, \dots, w_n)
\]
and a target value \(s \in \mathbb{N}\), the task is to determine whether there exists
a binary vector
\[
x = (x_1, x_2, \dots, x_n) \in \{0,1\}^n
\]
such that
\[
\sum_{i=1}^{n} x_i w_i = s.
\]
In this formulation, each variable \(x_i\) indicates whether the weight \(w_i\) is
included in the subset. The problem is NP--complete: verifying a proposed solution
is efficient, but no polynomial--time algorithm is known for solving arbitrary
instances.

\subsection*{Example of a general instance}
Consider the sequence
\[
W = (3,\, 8,\, 12,\, 19)
\]
and the target value
\[
s = 27.
\]
To decide whether a subset sums to \(27\), one must explore combinations of the
weights. In this case, a valid solution exists:
\[
8 + 19 = 27,
\]
corresponding to the vector
\[
x = (0,\,1,\,0,\,1).
\]
For a general instance, systematically finding such a vector may require testing
an exponential number of subsets in the worst case.

\subsection*{Superincreasing sequences}

A sequence \(W\) is called \emph{superincreasing} if every element is strictly larger
than the sum of all previous ones:
\[
w_i > \sum_{j=1}^{i-1} w_j \qquad \forall i \ge 2.
\]
For such sequences, the subset--sum problem is solvable by a simple greedy algorithm
that starts with the largest weight and works downward.

For a general subset-sum instance with an arbitrary sequence of weights, the
target value may have no solution, one solution, or multiple different
solutions. More importantly, without additional structure, finding a valid
subset may require exponential search in the worst case. In contrast,
superincreasing sequences have strong structure: each decision in the greedy
process is forced, so the solution (if it exists) is recovered efficiently and
deterministically. This property is the key reason why superincreasing
sequences are used as a private trapdoor in the classical Merkle--Hellman
construction.

\subsection*{Example of a superincreasing instance}

Consider the sequence
\[
W = (2,\, 5,\, 11,\, 23),
\]
which satisfies
\[
5 > 2,\quad 11 > 2+5,\quad 23 > 2+5+11.
\]
Let the target value be
\[
s = 28.
\]
The greedy procedure proceeds as follows:
\begin{enumerate}
    \item \(23 \le s\), include it: \(x_4 = 1\), new \(s = 5\).
    \item \(11 > 5\), skip it: \(x_3 = 0\).
    \item \(5 \le 5\), include it: \(x_2 = 1\), new \(s = 0\).
    \item \(2 > 0\), skip it: \(x_1 = 0\).
\end{enumerate}
The algorithm terminates with the unique solution
\[
x = (0,\,1,\,0,\,1),
\]
corresponding to
\[
23 + 5 = 28.
\]

\subsection*{Summary}

The essential feature of the knapsack problem is the strong contrast between
general instances, which appear computationally intractable, and structured
instances such as superincreasing sequences, which admit efficient solutions.
This distinction forms the mathematical basis for constructions discussed in
later chapters.

\section{Knapsack Cryptosystem}

The Merkle--Hellman cryptosystem was one of the earliest public-key encryption
schemes based on the subset-sum problem \cite{MerkleHellman1978}. It was
introduced in a model where secure communication does not require pre-shared
secret keys. Its design uses a superincreasing private sequence as a trapdoor
and publishes a transformed sequence that should appear computationally hard to
solve without the private key. Although the idea was influential, later
cryptanalysis showed that the basic construction is insecure \cite{Shamir1982}.
The following description focuses on the classical Merkle--Hellman construction
itself, namely key generation, encryption, and decryption.

\subsection{Key Generation}

The goal of key generation is to create two representations of the same
knapsack instance: one easy for the owner and one hard-looking for everyone
else \cite{MerkleHellman1978}. The process starts by choosing a superincreasing
sequence \(W=(w_1,\dots,w_n)\), which will serve as the private trapdoor. Next,
the key owner chooses a modulus \(m\) larger than the sum of all private
weights and a multiplier \(r\) that is invertible modulo \(m\). These values
are used
to hide the private sequence by modular transformation, producing a public
sequence \(B=(b_1,\dots,b_n)\), where \[ b_i \equiv r\,w_i \pmod m. \] 

In the classical description, a permutation is also applied to further obscure
the original structure. The public key is the transformed sequence (and
permutation, if used), while the private key consists of the superincreasing
sequence and transformation parameters \cite{MerkleHellman1978}.

\subsection{Encryption}
After key generation, the sender uses the public sequence \[
B=(b_1,b_2,\dots,b_n) \] to encrypt an \(n\)-bit message \[
x=(x_1,x_2,\dots,x_n), \quad x_i \in \{0,1\}. \] The ciphertext is computed
as \[ C=\sum_{i=1}^{n} x_i b_i. \] This means that only the public weights
corresponding to message bits equal to 1 are added. From the sender's
perspective, encryption is therefore straightforward and efficient, because
it only requires additions over public values \cite{MerkleHellman1978}.

\subsection{Decryption}
To decrypt the ciphertext \(C\), the key owner uses private parameters and
first computes
\[
S' \equiv r^{-1} C \pmod m,
\]
where \(r^{-1}\) is the modular inverse of \(r\) modulo \(m\). Because the
public weights were constructed from the private superincreasing sequence,
this step maps the problem back to an equivalent subset-sum instance over
the private sequence. The resulting value \(S'\) is then solved by a greedy
procedure, starting from the largest private weight and moving downward. At
each step, if the current weight does not exceed the remaining sum, the
corresponding bit is set to 1 and the weight is subtracted; otherwise the
bit is set to 0. If permutation is used during key generation, bit positions
are restored according to the secret key order after greedy recovery
\cite{MerkleHellman1978}.

\subsection*{Example}
Consider a private superincreasing sequence
\[
W=(2,\,5,\,11,\,23),
\]
with \(m=47\) and \(r=9\), where \(\gcd(r,m)=1\) and \(m>\sum W=41\).
The public key is
\[
B=(18,\,45,\,5,\,19),
\]
computed as \(b_i \equiv r w_i \pmod m\).
For message \(x=(1,0,1,0)\), encryption gives
\[
C=18+5=23.
\]
For decryption, \(r^{-1}\equiv 21 \pmod{47}\), so
\[
S' \equiv r^{-1}C \equiv 21\cdot 23 \equiv 13 \pmod{47}.
\]
Using greedy recovery on \(W\): \(13=11+2\), so recovered bits are \(x=(1,0,1,0)\).

\subsection*{Assumptions and Limitations}
Although the classical Merkle--Hellman design is efficient for encryption and
decryption, it assumes that the transformation from the private
superincreasing sequence to the public sequence sufficiently hides the trapdoor
structure. In practice, this assumption is not strong enough for the basic
construction: later cryptanalysis showed that equivalent private structure can
be recovered efficiently in polynomial time \cite{Shamir1982}. Therefore, the
original scheme is historically important, but not secure for real-world use,
which motivates the analysis of weaknesses and attacks in the next chapter.
